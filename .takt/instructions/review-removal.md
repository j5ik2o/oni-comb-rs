**レガシーコード削除の安全性**のレビューに集中してください。

**レビュー観点:**
- 削除対象の選定が妥当か（本当に不要なコードか）
- 削除漏れがないか（関連する死コード・テスト・ドキュメント）
- ビルド・テストが通るか
- 公開APIの互換性が維持されているか
- 削除範囲が適切か（過大でも過小でもないか）

**前回指摘の追跡（必須）:**
- まず「Previous Response」から前回の open findings を抽出する
- 各 finding に `finding_id` を付け、今回の状態を `new / persists / resolved` で判定する
- `persists` と判定する場合は、未解決である根拠（ファイル/行）を必ず示す

## 判定手順

1. 分析レポート（`{report_dir}/00-legacy-analysis.md`）と実際の変更差分を照合する
2. 削除された各項目について、呼び出し元ゼロであることを検証する
3. 削除漏れ（関連コード、テスト、コメント）がないか確認する
4. `cargo build --workspace` と `cargo test --workspace` の結果を確認する
5. ブロッキング問題が1件でもあれば REJECT と判定する
